## 0. Essence

**A shared mind that turns declared goals into real-world outcomes, through any chatbot, forever.**

Four jobs — hold goals; hold the mind; execute; compound. Everything else is plumbing. The product is a memory-and-goals layer for the chatbot era with an execution engine attached.

Cosmic ambition stated openly, always grounded in the current milestone: open coordination of humans + AI on goals previously thought impossible (world hunger, multiplanetary life, shared fantasy universes, games) — "we all accomplish together, not waiting on anyone." Real-world action feeds back as evidence; gates release; larger goals decompose and resolve.

**0.1 The canonical scene** (the defining UX; requires merge-on-auto, §7): a user's chatbot, building something novel, hits a platform capability gap. It files a patch request — because the fix helps its user's project. While STILL IN THE SAME USER TURN, working between replies, the patch goes through the loop: implemented, merged, deployed, an implementation note written onto the very branch the chatbot was building, plus real-world-feedback user-test verification so the chatbot knows how to report results (initial and ongoing). The chatbot finishes its design inside the same reply. Skimming users never learn there was a gap — they just see it working. The patch loop is today's small-fast-patch example; every loop investment is measured against this scene.

Five standing design principles: (1) **Appreciation test** — would it survive deleting the code and rebuilding on next year's models? Brain, goals, ladders, lessons, attribution survive; scaffolds don't. (2) **Carry test** — the brain carries the intelligence so cheap models can run the loop; smart-model hours are for design and curation. (3) **References over copies; lenses over containers; assembly over storage** — knowledge is never duplicated into a scope; the system points and assembles. Forking is the deliberate exception: an intentional copy WITH lineage. (4) **Build-boundary law (host 2026-06-10)** — the platform builds ONLY substrate: if users could compose it from primitives, we must NOT build it in code; we ship forkable defaults as branches and wire in capability gaps only (new primitives, effector/sense adapters, state enforcement, security boundaries, the engine itself). Every component carries a [substrate]/[composable] verdict (§13 audit); building a [composable] in platform code is drift. (5) **The basis quest (host 2026-06-10)** — the workflow-building tool surface exists to DISCOVER and FORMALIZE the minimal complete primitive basis for composing all workflows: computing's tiny-instruction-set story (C++ → C → a handful of operations), replayed for work itself. The basis stays the shortest possible, extremely powerful list — current candidate: Node/Edge/State/Scope/Run/Trigger + 5 handles — formalized, frozen-small, and open-source commons, because a formalized basis is infrastructure for everyone or it is nothing. Convergence audit (2026-06-10, primitive-basis-audit; codex verdict 2026-06-10: **ADAPT** — six names locked as working vocabulary, contract details stay open per the audit's open questions): **six confirmed, none added, all six amended with contracts** — CLAIM reducer + SEAL on State, structured cancellation + suspended continuations + terminal-event guarantees + identity dedup on Run, failure contracts + completion tokens on Node, join modes on Edge, capability recast on Scope, run-creating slimming on Trigger — plus state laws L1–L12 and two forever-tests (the McCarthy artifact: a meta-circular branch that interprets any branch using only the six; and a pattern scorecard with encoding-overhead budget). All ops-layer concepts then COMPOSE as library branches: Forth's discipline — what is buildable from the kernel stays out of the kernel. The basis speaks in two registers (§0.2).

**0.2 The two-register vocabulary (host-approved 2026-06-10).** One language, two registers, both canon. The **kernel register** — Node, Edge, State, Scope, Run, Trigger — is the structural grammar: graph definitions, substrate code, formal docs, the McCarthy artifact. The **spoken register** is how every user-facing surface talks (voice, tool descriptions, chatbot narration); each spoken term has a precise kernel meaning, DISCOVERED from what builders and users already say — never invented (naming census frozen in the basis-audit appendix, per codex adaptation #1):

| Kernel | Spoken | Precise meaning |
|---|---|---|
| Node | **step** | a step is a node's execution inside a run ("failed at step 3" = that node's execution). Resolves the field's own split — graph dialects say node (LangGraph, n8n), runtime dialects say step (Inngest, Cloudflare Workflows, GH Actions): definition vs. execution, both right. |
| Edge | **then / after / if-it-fails** | spoken edges; "connection" for the drawn artifact (n8n's word). |
| State | **state**; **memory** when brain-backed | run-scoped state vs. brain entries — surfaces keep the distinction explicit. |
| Scope | **permissions / "scoped to"** | OAuth's gift (RFC 6749 §3.3); capability set on handles. |
| Run | **run** | exact hit — already the field's universal word. |
| Trigger | **trigger / "when X happens"** | exact hit — Zapier/n8n/GH Actions all say trigger verbatim. |

The chatbot is the compiler between registers (§9 user-vocabulary discipline: speak spoken, write kernel). New spoken terms enter only by amendment carrying census evidence of real usage; the kernel register changes only through the basis quest (law 5).

## 2. The Asset (core concept)

Progression: **idea → goal → branch (under construction) → published → running → ASSET.**

An **asset** = a branch/gate/goal loop that is (a) coupled to the real world, (b) receiving feedback from real events, (c) verifiably producing the owner's desired output toward a wider goal. A payables workflow is a branch until real companies run real work through it with tracked success metrics (accountant approvals, shared by consent) — then it's an asset. Tiny posting → engagement → behavior adapting = asset #1 ("producing your will in the real world"). Assets are the platform's success metric (count + health) and the eventual economic surface ($tiny attaches to assets — frozen). Demos must mint assets, not branches.

## 3. The Mind anatomy — and universes as seeds

Organs: **soul** (will + governance) · **brain** (knowledge) · **goals** (intents) · **skills** (branches) · **voice** (presence) · **hands** (effectors) · **senses** (inbound adapters — reads the world; see §6.1). Tiny = mind #0 whose org is the platform, dogfooding how anyone personifies their own intelligence.

**Every universe is a seed, and a seed is a guide — not a starter kit.** Seeds derive their shape from the evolving universes. A seed contains no branches of its own: it carries REFERENCES into the commons ("knowing where to look"), because the user's chatbot already has access to every commons branch through the lens of its user's objective. The chatbot easily sees what others in similar universes do for similar goals — so when the user wants already-done-before things, **assembly is instant**; building happens only at the frontier. The seed's whole job: guide the chatbot to build its universe's brain FROM the commons brain — assembling useful branches, goals, sub-goals, tested assets, and gates through the objective lens. The seed is versioned and commons-improved; `create_universe` mints the current seed; "summon your own mind" = create_universe, polished. Existing universes are the trees (grown before the seed was designed).

## 4. One actor model

Actors: **human** (incl. host), **chatbot session** (any provider; "user" and "contributor" are the same type), **daemon** (24/7 capacity), **simulated user** (first-class persona; powers the experience canary). All actors query the context engine through their own lens, claim work, contribute lessons, accrue attribution.

**Humans are claimable actors** (the world-coordination primitive): await-human nodes (task + evidence contract), a human claim surface, real-world evidence intake via gate_events (attester ≠ verifier; substrate exists, dark). AIs automate the automatable; humans claim the rest; the brain remembers what either learned.

## 5. Brain v2 — the multi-lens context engine (centerpiece rebuild)

**The brain is one big multi-lens, custom-in-the-moment context engine.** Chatbots already have their own objectives from their users — the platform never needs to know or model them. There is NO attach ceremony and no objective registration: the chatbot brings the lens (its own description of what it's doing, plus optional structured hints — goal ids, universe, shape tags, budget), and the brain assembles the view on the fly. A goal is a mind; a universe is a mind; a session forms a transient mind through its lens — drawing thoughts from anywhere useful in the entire commons mind — contributes lessons/filings back, and dissolves.

- **Primary verb: `assemble(lens) → view`.** Deterministic core assembly (cached), relevance-ranked: exact scope + related goals + similar-SHAPED objectives (goal-graph edges, shape/tags, embeddings — PR-139's tag-matrix + vector substrate finally get their consumer). Discovery and remix dissolve into this verb — "what are others doing for similar goals in similar universes" is just a lens query; no separate discovery surface.
- **Many brains, one commons.** Each universe keeps its own brain (sovereignty, trust tiers, self-host parity). Entries are goal-addressed; the commons = union of public, goal-addressed knowledge across universes.
- **Entries**: typed (fact, lesson, decision, direction, design, filing, claim, receipt, trace-summary…), scoped (goal/universe/visibility), lifecycled (candidate→accepted→promoted→superseded), TTL-classed (knowledge persists until superseded; coordination auto-archives when its work completes — kills sprawl structurally).
- **Hard size caps everywhere; size pressure — never clock time — drives forgetting (host 2026-06-10):** every brain tier carries a hard size cap — pinned slots (bytes), views (tokens per host), candidate pools, recall-trace logs, snapshot history, per-universe totals. Lifecycle events (consolidate, squash, archive, degrade-to-reference) fire on cap pressure (e.g. consolidate-at-80%, Hermes' proven mechanic, generalized): inactive universes lose NOTHING; active ones consolidate in proportion to their own growth. Caps are per-universe config — seeded defaults, org-chart governed, surfaced in status, cheap to test and retune as size matters more or less per tier. Existing substrate: the BUG-023 storage-caps machinery (WORKFLOW_CAP_*, warn/critical pressure levels).
- **Views are budgeted; inclusion is competitive** (relevance × promotion × recency × link-density). Pinned slots where a soul/goal is in scope: hard lines, safety rules, current direction. What doesn't make the view effectively doesn't exist — verbose chatter self-defeats. The budget IS the attention economy.
- **Smart models write, cheap models read**: LLM intelligence happens at curation time (consolidate/supersede/summarize) and is STORED; assembly itself never pays per-recall LLM cost. **Curation is a user-buildable skill** — default curator ships with the seed; orgs fork it; commons moderation = curation at commons scope, org-chart routed.
- Store: SQLite canonical (entries + FTS + vectors); wiki becomes a RENDERING of document-type entries (stable URLs); nightly git snapshot for durability/offline. STATUS.md, agent memories, ideas files become views; brain canonical. Index-is-disposable invariant: FTS/vectors rebuild from entries in one command; chunk identity = hash(source:lines:content:model_version).
- **Research-validated mechanics (2026-06-10 study — see the brain-v2 research companion; codex ADAPT 2026-06-10: design stubs + read-only assemble() unblocked, six pre-build gates in `docs/audits/2026-06-10-tiny-first-principles-codex-review.md`):** (a) **keyed supersession + bi-temporal validity** — topic-keyed fact/decision/direction entries auto-supersede at accept; default views NEVER contain superseded entries; lineage stays queryable; (b) **recall traces + usage-evidence promotion** — every assemble() logs what it served; promotion is earned by inclusion across DIVERSE lenses, not writer judgment; (c) **RRF multi-channel assembly** (FTS + vector + exact-key + temporal filter → rank fusion → our score as rerank) + 1-2-hop link activation from top seeds; (d) **write-time contextual enrichment** — curation prepends a 50-100-token situating paragraph per entry (-67% retrieval-failure recipe); (e) **progressive-disclosure views** — pinned (hard byte caps) → promoted rollups → manifest of id+one-liners → fetch verb; under-fill ~60-70% of budget; budget-losers degrade to references, never disappear; (f) **recitation slot** — per in-flight goal, an auto-maintained current-plan/progress/next-step block (the cheapest "smarter on complex work" lever); (g) **delta verb** — assemble(lens, since=cursor) returns only promotions/supersessions/new entries; (h) **HTML as a rendering+addressing layer** — entries stay canonical, markdown stays the authoring interchange, HTML renderings carry stable element IDs for fragment-level citation and direct browser/surface serving.
- Instrumented bets: **view hit-rate**, **tokens/query** (target 2-7K default views; ~170-token minimal wake-up tier), **assembly latency** (p95 <200ms bar), **fresh-session onboarding cost** (target: minutes/<50K tokens; 2026-06-09 baseline was a 3.5M-token archaeology dig), plus a lens-query eval harness (NDCG/MRR + temporal probes, nightly) built BEFORE weight-tuning. The context engine is THE hard artifact of this spec — everything else is composition; this must be measurably good.

### 6.1 The operations layer — how a mind pursues goals across months of real time

The engine produces minutes-long runs; goals live for months with real-world events arriving on their own schedule. Between run and goal sits the **operations layer** — the patch loop was its first instance, never named as such. A pursued goal is a **program**: goal-tree + ladder + steward + staffed stations + senses + hands + brain + caps, advancing at each surface's patch boundary.

- **The steward (the patch loop, generalized).** Every actively-pursued goal has a standing loop branch — its steward — that wakes on rhythm or events, reads the goal's recitation slot, decides what belongs in flight, dispatches work, files lessons, sleeps. `soul.loop_branch_def_id` is the universe-level steward; goals get the same pattern. Stewards are user-built and forkable.
- **Standing waits (vigils).** Runs never wait; PURSUITS do. A vigil is a stored condition→dispatch pair owned by a goal: "when evidence E arrives / deadline passes / sensor fires → dispatch branch B with this context." Weeks-long real-world waits (test readers reading, Valve reviewing, players playing) become durable continuations, not open runs. Substrate half-exists: scheduler event subscriptions + dispatcher.
- **Senses — the seventh organ.** Hands write to the world; senses READ it: standing inbound adapters (Discord/forums, email, webhooks, store reviews, sales/telemetry feeds) turning external streams into typed, provenance-stamped candidate entries and events — the gate/consent pattern of effectors, mirrored inbound (volume-deduped, curation-summarized). Prototype: the Echoes Reddit intake branch. Without senses, "real-world feedback" means a human pasting; with them, the feedback half of every asset loop exists at volume.
- **Staffing — the trust ratchet generalized.** Every node/station in a branch is STAFFED: automation, a named human, any community member holding capability X (a claim), or hybrid (auto-draft + human-approve). Staffing is per-goal, owner-edited: each org defines which parts humans do vs automate — the prose author drafts herself and automates continuity checks; the studio auto-suggests balance changes and humans approve gameplay. Re-staffing stations human→auto as evidence accrues IS the trust ratchet; merge-code was just the first station.
- **Playbooks — forkable goal-trees.** Decomposition is user-built (planner branches propose sub-goals + ladders; the org chart adopts) and REUSABLE: a proven goal-tree + ladder + staffing defaults + branch bindings = a playbook in the commons. Wedding → conference → startup-launch rhyme because they share shape; forking a playbook instantiates a program in your universe. Seeds reference playbooks ("knowing where to look"). Substrate half-exists: goals define_protocol, gate ladders, the HTN planner.
- **Resource caps.** Spend (LLM, infra, bounties) obeys the same law as memory: hard caps per goal/universe, pressure-driven decisions, surfaced in status (cost-ledger substrate = the PR-172 work).
- **Artifacts stay yours (doctrine).** The product — game repo, manuscript, builds, assets — lives in the org's OWN systems; the brain holds references, receipts, provenance. The platform is the nervous system around your tools, never the warehouse. (References-over-copies, applied to the product itself.)

Worked example, game studio: their repo stays theirs; players file via senses (Discord → filings); the patch loop runs against THEIR repo; a release-train steward holds the rhythm; patches land at the next game round (patch boundary); designers staff the gameplay-approval stations while balance suggestions automate; telemetry senses feed evaluators; the ladder runs playable → vertical slice → beta → Steam. Worked example, serialized author: weekly-story steward; feedback senses; community canon behind gates; test readers = await-human evidence; editors = staffed stations; publication = effector + gate; every book's lessons compound into the next via the brain.

### 6.2 The failure layer — when programs go wrong for weeks

- **Health is evidence, not silence.** Every program emits a derived heartbeat — steward last-woke, dispatch claim rate, ladder delta, sense inflow. The uptime canary was the first program health probe (the dogfood built this organ too); PR-172 telemetry is the substrate.
- **Typed program states, derived not declared:** active · degraded (runs failing) · starved (dispatches unclaimed) · sick (steward not waking) · **hollow** (activity without ladder movement — the 980-runs-on-6-bugs lesson, detected via ladder-delta/activity ratio) · **dormant** (sleeping with vigils armed — HEALTHY: per the size-pressure law, inactivity costs nothing and loses nothing) · parked (explicit) · abandoned (succession fired) · completed.
- **Watchers terminate in code and humans, not turtles.** Cheap deterministic probes (canaries) → the universe steward → the soul's declared escalation ladder (who gets told at which silence threshold; seed default: probe → steward → founder ping → commons patrol lens for public goals). Programs may hold vigils on sibling programs — community cross-watching is the redundancy.
- **Escalation is staffing in reverse.** An automated station failing repeatedly escalates to human staffing; a human station timing out escalates per the org chart (another human, the founder, or lower-authority automation). The trust ratchet runs both directions — failure is just downward evidence.
- **Cascades are declared.** A program holds vigils on the upstream assets it consumes; upstream death fires downstream vigils ("your input died") instead of silent starvation.
- **Abandonment + succession.** The soul holds succession entries (SUCCESSION.md made per-universe soul state). A founder silent past the escalation thresholds triggers the abandoned path: public goals are offered for commons adoption — adopt or fork the playbook; work is never orphaned (no-lock-in includes no-lock-DEAD). Private universes get a grace period → archived with full export. Assets with real users get continuity priority.
- **Failure compounds.** Every program failure files lessons against the playbook that spawned it — the postmortem→probe ratchet generalized: playbooks accumulate immune systems, so the second org to fork one inherits the first org's scars.

### 6.3 The attention layer — what decides which goal gets capacity

The scarce resource at every layer of this system is the same thing: **attention** — machine capacity (daemon/LLM time), human attention (staffed stations, keys, evidence), public attention (discovery, voice), and memory attention (view budgets). One law governs all four: **attention is budgeted (hard caps), earned (signals), allocated (owned policies), and receipted (attribution).**

- **The platform never allocates; it matches.** No central planner. Demand side: programs emit **work offers** (claims, filings, stations, bounty-tagged tasks) onto a unified offer surface — the dispatcher queue generalized across goals (blackboard-pull beats push, 13-57% in the research). Supply side: every unit of capacity has an OWNER whose soul declares an **attention policy** — a user-buildable selector over the offer stream ("my org first, then allied goals, then highest-signal starved work, then curiosity"). Today's hardcoded dispatcher tier weights (host 100/owner 80/user 60/paid 50/pool 40/opportunistic 10) become the seed policy: the first forkable default, not the law.
- **Policies compile to deterministic scorers** (carry test): smart models WRITE policies at curation time (weights, rules, signal expressions); pick-time evaluation is cheap and deterministic — never an LLM call per task. Same shape as view assembly.
- **Two allocation regimes, one surface:** OWNED capacity is steward-directed and hierarchical — a program's steward allocates its budget downward, caps cascade (goal caps → sub-goal caps); "which sub-goal gets capacity" is decided at the goal's own level, recursively (subsidiarity). COMMONS capacity is volunteer matching — idle capacity browses the offer stream through its soul's lens ("idle is a failure state" → the opportunistic tier, finally real).
- **The signal vocabulary is the price system before money:** ladder delta (progress), asset status (real-world coupling), hollow/starved flags (§6.2 — capacity flows AWAY from hollow programs toward working ones), vigil-density (how many programs depend on this — criticality), cosigns (human endorsement), staffing starvation, recency, and — when $tiny unfreezes — bounty, which plugs into this slot as just another signal, changing nothing architecturally. Incentives affect PICKUP, never acceptance/merge (locked doctrine).
- **The platform owns signal INTEGRITY, not allocation policy.** Signals must be ungameable: receipts behind every signal, evidence behind every ladder claim (attester ≠ verifier), provenance everywhere, recall-trace forensics. Derived doctrine: **discovery ranking is never for sale** — selling placement would corrupt the one thing the platform must keep honest (and is already implied by no-lock-in: never route users toward profitable lanes).
- **Human attention is the scarcest and best-protected:** never demanded, only requested-with-signals; giving it (review, approval, evidence) mints contribution events → attribution → reputation/bounty. Escalation budgets + paging ladders keep founders' attention from being the system's default fallback.
- **Anti-capture properties are structural:** policy diversity is automatic (policies are per-owner souls — no monoculture formula to capture); rich goals can't buy capacity whose owners' policies ignore money; spam offers die in candidate quarantine + quotas (§11.1); demand shocks degrade to honest queueing (§12). Tiny's own attention policy can deliberately counter-program: serve the starved, narrate the unglamorous.
- Substrate already present: dispatcher queue + tier scoring (becomes policy-driven), pickup_signal_weight/effort lanes, soul-guided dispatch hook (coefficient 0.0 today — becomes the policy binding), NodeBid producer (frozen), subscriptions, contribution_events.

## 7. Soul & the org chart (deliberately under-engineered v0)

**Governance lives in the soul and is designed from observed error, not up-front schema.** v0 ships only: (a) **founder recognition** — OAuth/GitHub identity binding (the PR-165 actor-identity substrate makes this nearly free); (b) the soul **free-texts its understanding of its org** — who it serves, what it may decide, higher-level ideas with freedom rather than strict rules; (c) a **governance wrongness log** — every authority-relevant decision records {scenario, soul's interpretation, outcome, founder override?} as lessons on the governance goal. Since everything consequential (merging, spending) is blocked by the host key anyway, getting the org chart wrong is cheap — **how it gets it wrong across scenarios IS the design signal**, and the routing-table schema (decision classes → {autonomous|role|human|vote} + evidence ratchets) remains the north-star sketch we grow into only as the wrongness corpus demands it.

Seed routing intent (host): merge-code + spend-money → host, slowest ratchet. Publish-post, edit-soul, moderate, create → delegate early ("fun experiments"). Full autonomy is the design target — the canonical scene (§0.1) requires autonomous merge in safe classes; the human key is routed, not removed.

## 10. Seed-grown platforms & patch boundaries

Doctrine: **platforms of the future won't be built — they'll be grown from seed**, evolving in real time around their seed users/founding team. The patch loop was always a dogfood problem teaching the substrate how to build this.

**Patch boundary principle**: every surface has a natural boundary where evolution lands — a chat turn (minutes; the canonical scene), a page refresh, a **game round** (the Unreal RTS/FPS via Steam: next match loads the host's latest patch), a session. The platform targets each surface's NATURAL boundary; no universal latency requirement.

**Surface definition** primitive: premade UX shells (chat, site, wiki view, game canvas) render entirely from universe state; the mind evolves DATA in real time; code-gen evolution stays on the slow path (a constrained, validated world-spec DSL makes "the room changes as you talk" safe and fast). Staircase: (a) seed → shell bound to a universe (the website is already rung (a) — it renders from MCP state); (b) hot-swapped surface state (minutes); (c) world-spec DSL surfaces at conversational boundaries (seconds); (d) **the White Room** — multiplayer first-person blank construct evolving around its founding players as they talk. North-star demo; game-factory + self-evolving-website lanes are the staircase. New infra flagged honestly: real-time presence/fan-out (websockets) for (c)/(d).

## 11. Universes, orgs & trust tiers

Default: universes are **open** (commons). As the soul learns its org, parts may convert to **permissioned-cloud-private**, **local-private**, or **self-hosted** — per that org's own org-chart concept. North-star: compliance-grade provable cloud privacy (proprietary/HIPAA-class) so orgs never NEED to self-host but always COULD; hybrid seamless. Near-term build: open + local-private; design seams now. **Supersedes prior commons-first absolutism** in PLAN.md.

**No-lock-in doctrine (invariant):** fully open source, buildable + customizable by all the world's nerds; export-everything; capabilities never gated toward profitable lanes; if the platform can't do what someone needs, it evolves so they can pursue their vision their own way. Self-host parity continuously probed.

### 11.1 Commons governance (no new system — three existing primitives)

The commons is governed by the SAME primitives as everything else: **lifecycle** (candidate quarantine = the security boundary), **propose/adopt** (anyone proposes; trusted actors adopt), **org chart** (who is trusted, per scope — commons scope routes through Tiny's soul; each universe's brain through its own).

- **Identity tiers, zero-friction preserved:** T0 anonymous session actor — may file + create candidates (harmless by construction: candidates never enter other minds' default views). T1 durable pseudonym (host OAuth subject when present — PR-165 substrate) — accrues a track record; capabilities (cosign, propose-supersession, curator rights per scope) unlock by contribution EVIDENCE, not payment or paperwork. T2 verified founder (GitHub OAuth) — org-chart authority over their universes. Ratified 2026-06-10: T0 = file + candidates only; cosign requires T1.
- **Supersession authz (the vandalism guard):** supersede your own entries freely; for others' entries you file a **proposed supersession** — a citation-required candidate contradiction link — adopted by a curator skill, usage evidence, or the org chart. Authority lives at adopt, never at propose (platform doctrine applied to memory). Contested pairs render with contest markers and route to curation; the bi-temporal store makes every act reversible (rollback = re-accept).

### 11.2 Redaction — the right to disappear from a never-delete store

- **Prevention first:** secrets/PII scanner at the candidate gate (block with redirect: "this looks like a secret — not stored; rotate it if real"); the anti-collision profile-rejection already filters most personal data before it exists.
- **`redacted` is a distinct terminal state, NOT supersession:** body physically deleted everywhere; a tombstone keeps id/type/reason-class/authority/timestamp/content-hash; inbound links render "[redacted: class]"; bi-temporal queries return tombstones, never content. Redaction is the one sanctioned exception to references-over-copies: copies are chased.
- **Propagation pipeline (ordered):** entry body → FTS/vector indexes (disposable-by-design — index rebuild becomes the compliance feature) → rollups/consolidations that cited it (recall traces + link graph locate exactly which; re-curate or mark stale) → wiki re-render → git snapshot.
- **Git snapshot policy:** the snapshot is a mirror with reserved force-push, not a collaboration repo. Current-state files drop content at the next snapshot commit; history squashes when a per-universe **history size cap** is exceeded (size pressure, not clock time — a dormant universe keeps its full history forever; a busy one squashes in proportion to its own growth), bounding how long history remembers; urgent class (secrets) = immediate filter-repo + force-push AND rotate the secret (rotation is the real remediation). Crypto-shredding (per-entry keys; delete the key to erase) is reserved for permissioned-private tiers where snapshots hold ciphertext anyway.
- **Authority + transparency:** redaction routes through the org chart (ratified 2026-06-10: day-1 = founder, plus self-author within a grace window); every redaction emits a PUBLIC receipt (who/reason-class/when — never content) — the Wikipedia deletion-log precedent; counts surface in status transparency.
- **Honest limit, recorded:** content already absorbed into host chatbot memories is beyond reach; freshness-stamped supremacy headers + fresh views stop future spread, and the policy says so plainly.

### 11.3 The compliance layer — regulated industries (HIPAA as archetype)

Every regulated industry has the same five objects: a **rulebook** (external regulation), **SOPs** (internal policy), **evidence requirements**, **auditors** (external attesters), and **change control** (rules and tools change). The platform's wedge is structural: everywhere else, the SOP binder and actual practice drift apart and the auditor's job is finding the gap — here **the SOP is executable (the branch IS the procedure) and execution self-documents (runs ARE the evidence)**. Audit stops being archaeology and becomes replay. Per the build-boundary law, compliance CONTENT is community-built; the platform builds only the evidence primitives that make compliance content provable.

**Map to primitives:** rulebook → versioned, cited commons entries with effective dates (bi-temporal: "what did the rule say when we ran this"). SOP → playbook (the compliance procedure is a forkable branch). Authority matrix → org chart; segregation of duties / 4-eyes → staffing constraints + attester≠verifier (van der Aalst resource patterns cover this natively). Access log → the journal (L6: every effect AND every cross-scope read journaled; "private chats, public actions" ledger). Validation → acceptance scenario packs + autoresearch (continuous re-proving); banking's SR 11-7 model-risk trio (inventory/validation/monitoring) = branch registry / scenario validation / hollow-asset monitoring, exactly.

**[substrate] — the evidence primitives to get right:** (a) journal completeness incl. **model/provider version stamps on every run receipt** (needed for both audit and model-change revalidation); (b) **auditor as actor**: time-boxed, scoped read capability + an auditor lens whose **evidence-packet export carries an integrity guarantee** (hash-chained receipts; packet format itself is composable); (c) **legal hold + retention floors** — the inverse of size caps: records that must be KEPT n years get a hold flag that overrides redaction (except statutory-erasure classes; jurisdiction conflicts are arbitrated by the org chart, policy content is composable); (d) **synthetic-data effect handlers** — the Scope-as-effect-handler recast pays off: the same branch replayed against a synthetic-data interpreter = validation and auditor walkthroughs without ever touching PHI; (e) crypto-shredding + residency in the private tiers (§11); (f) **regulator-feed senses** as a named sense class (HHS/Federal Register/etc.).

**Change control = the cascade machinery (§6.2), nothing new:** a regulation update or a NEW MODEL RELEASE are both upstream-dependency changes. Sense fires → vigils on dependent playbooks trigger → stewards dispatch re-validation (pinned scenario suites re-run via autoresearch) → remix proposals mint → orgs adopt at their own patch boundary → canonical swaps only on validation parity. Regulators and model vendors are just upstream assets your programs hold vigils on.

**The compounding story (early adopters → everyone):** a brave clinic + consultants build the HIPAA-intake playbook → validates on scenarios → passes a REAL audit → that outcome is gate evidence on the playbook's own ladder: drafted → scenario-validated → adopted-by-N-orgs → **survived-real-audit** → cited-by-regulator. Fifty clinics fork the proven playbook; every fork's validation adds evidence; autoresearch re-proves it on every model generation. The commons accumulates certified-grade process knowledge no single org could afford — compliance becomes shared infrastructure (the asset ladder applied to compliance content).

**Honest posture:** we never claim certifications we don't hold. Near-term truth: self-host gives regulated orgs full control today (no BAA with us required); platform-side compliance tiers (BAA-grade cloud, SOC 2) are north-star organizational work, staged per §11. The NOW work is completeness of the evidence primitives — so that when serious orgs arrive, the substrate already proves everything it does.
