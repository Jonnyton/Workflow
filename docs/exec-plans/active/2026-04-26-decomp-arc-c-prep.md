# Arc C (env-var deprecation alias deletion) — pre-flight scope

Read-only scope for deleting two legacy env-var aliases in `workflow/storage/__init__.py`:
- `UNIVERSE_SERVER_BASE` → canonical `WORKFLOW_DATA_DIR`
- `WIKI_PATH` → canonical `WORKFLOW_WIKI_PATH`

**The smallest cleanup arc by LOC** (~25 LOC of resolver branches die) but with the **largest test-surface caller-migration** (~141 `UNIVERSE_SERVER_BASE` references + 16 `WIKI_PATH` references across ~60 files). Like Arc B, the audit's "1h" estimate undershoots because of the test caller burden.

**Recommended split:** Phase 1 (top-level code: `fantasy_daemon/__main__.py`, `packaging/mcpb/server.py`, `workflow_tray.py`, `workflow/api/engine_helpers.py`, `workflow/api/helpers.py`, `workflow/storage/__init__.py` itself) — ~6 sites + the resolver. Phase 2 (tests/ migration: 58 test files using `monkeypatch.setenv("UNIVERSE_SERVER_BASE", ...)` + 4 test files using `WIKI_PATH`). Phase 3 (resolver-branch deletion + plugin mirror sync).

## 2. Symbol enumeration — what dies

### 2.1 `data_dir()` resolver — `workflow/storage/__init__.py:210-222`

```python
legacy = os.environ.get("UNIVERSE_SERVER_BASE", "").strip()
if legacy:
    _reject_windows_path_on_posix(legacy, "UNIVERSE_SERVER_BASE")
    if os.environ.get("WORKFLOW_DEPRECATIONS", "").strip() in {"1", "true", "yes"}:
        warnings.warn(
            "UNIVERSE_SERVER_BASE is deprecated; migrate to "
            "WORKFLOW_DATA_DIR. Both currently resolve to the same "
            "path; UNIVERSE_SERVER_BASE will be removed in a future "
            "release.",
            DeprecationWarning,
            stacklevel=2,
        )
```

## 5. Risk profile

### Phase 1 (top-level code migration) — LOW risk

- Mechanical sed; the legacy alias resolver still works during the migration so green-state holds.
- Loud failure mode: typo in env-var name → test failure at first run.
- One tray-launcher coupling: `workflow_tray.py` env-set affects daemon startup; verify tray launch in dev environment after the edit.

### Phase 2 (tests/ migration) — LOW risk

- Mechanical sed at scale (135 sites). Higher chance of typo in one of 60 files.
- Mitigation: per-file `pytest tests/<file>.py -q` before moving on.
- `tests/conftest.py` migration affects ALL tests. Migrate conftest LAST + run full suite immediately after.

### Phase 3 (resolver deletion) — LOW risk

- The legacy-test function deletion is purpose-built (those tests verify behavior that's about to be deleted).
- Plugin mirror: `python packaging/claude-plugin/build_plugin.py` after deletion to propagate.
- External callers' env-vars: host's actual environments (CI secrets, deploy scripts) need to be scrubbed for `UNIVERSE_SERVER_BASE` / `WIKI_PATH`. **This is the one external-touchpoint risk** — env vars set outside the repo's reach.

### High-risk (HOST ENVIRONMENT)

- **Host's CI / GitHub Actions secrets / `$HOME/workflow-secrets.env` / DigitalOcean Droplet env-files.** If any of these still set `UNIVERSE_SERVER_BASE=...` after Phase 3 ships, the deploy silently misses config. **Mitigation: host scrubs all 4 surfaces BEFORE Phase 3 deletion lands.** Decision ask in §8.
